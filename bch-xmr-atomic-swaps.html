BCH<>XMR Atomic Swaps

Posted by
Rucknium
10 months ago
Description

Recently BTC<>XMR atomic swaps have become available on mainnet. Let's do the same for BCH!

The main challenge to implement BCH<>XMR atomic swaps is the transaction malleability issue. At least one BTC<>XMR atomic swap implementation leverages SegWit, which is BTC's solution to bitcoin's transaction malleability issue. BCH famously does not have SegWit. Rather, BCH tackled transaction malleability through what one BCH developer has termed a "whack-a-mole" strategy.

Relevant discussion of the transaction malleability issue is here, with one researcher believing that BCH<>XMR atomic swaps are possible without a protocol change to either blockchain:
https://bitcoincashresearch.org/t/monero-bch-atomic-swaps/545/2

The bounty should pay out if any open source implementation of BCH<>XMR atomic swaps is successfully demonstrated on their mainnets.

The only distinction here is the specified BTC scripts need to be updated to use Schnorr signatures instead of an on-chain multisig. The easiest way to do this would be to simply fork an existing atomic swap tool, of which there exist 3 in working condition. My own, https://github.com/MerosCrypto/asmr, one by tecnovert, https://github.com/tecnovert/xmrswap, and COMIT's https://github.com/comit-network/xmr-btc-swap/. I'd only recommend working with the former or the latter, and if you want this to extend beyond a PoC, the latter.

I can't understand the xmr-btc.pdf, specifically section 2.3 'Worst case scenario'.
Alice is buying BCH
Bob is buying XMR
It seems to me the worst case scenario occurs if Bob sends BCH 1st, because Alice can just scam him by revealing her XMR key share without ever sending XMR. Whoever sends money first can get in trouble, but the situation is less bad for Alice, since at worst her XMR is burnt, not stolen. It just looks like there's a first mover disadvantage.

I'm interested in a completely different solution, which we might also want as a plan B (backup). This doesn't count towards the bounty. We can have at least 8 well-known validators, then Bob has to move first, sending BCH to a contract which is determined by the 8 he's chosen. At least maybe 4/8 sigs are needed to forward the payment to Alice, minus the fee. No sig is required to refund, after the deadline. The BCH Script is revealed to all 8, as well as Alice. It could contain:

  • The 8 public keys (4/8 multisig).
  • Alice BCH forward address.
  • Bob BCH return address.
  • XMR public spend key (Bob).
  • XMR private view key (Bob).
  • XMR required payment amount (cumulative minimum).
  • Deadline (e.g. 6 BCH blocks ~ 1 hour).

The contract mustn't allow anything other than either forward or return payment. The 8 "custodians" can't possibly do anything else. (Edit: Alice needs to be able to instantly cancel the contract unilaterally, without waiting an hour for a new price.)

If Alice is happy with the 8 chosen & amounts (price), she sends XMR to Bob's address he's revealed in the BCH Script. If not, Bob gets a refund after an hour or so. Then the validators sign the forward payment when they are convinced XMR payment has been made.

Both Bob & Alice might use an Electron-Cash plugin to encode/decode the Script, & communicate with validators. The validators might run some server version of the software.

Bob could get spammed endlessly, unless he requires a tip in advance from Alice, to prove she's serious! Another issue is that we could be more clever with the Script in the long run, because all 8 PubKeys etc is a lot of data, and the Script itself only strictly needs a hash of a future data push, along with all raw opcodes as well, to make the system more efficient. I wonder if anyone is interested?

Edit: A couple months later I just think of an idea. Before Alice moves XMR to a 2of2 address, both Bob & Alice can sign a reneging XMR transfer using unlock_time in an hour. So Alice can reneg on XMR, in an hour, before she provides the 1st (XMR) transfer. She'd then have to reveal her private key share, within half an hour, on the BCH chain, to get BCH. However that might require a whole new BCH OpCode, like OP_CHECKPRIVKEY, which checks her revealed secret is correct. Bob's smart contract can only be sure of Alice's PubKey, constructed using her private key share from XMR.

If Alice doesn't send XMR, Bob would know thanks to the view key publicly shared in advance. Alice would never be able to scam Bob of his BCH. If that was possible, this wouldn't be atomic. Having your XMR burnt is equivalent to having it stolen, if not even worse for you if it's truly burnt, as there's no chance of recovery.

Whoever sends first, if the other party isn't serious, is at risk of losing their fees. This is natural to atomic swaps with the only solution being for Alice to cover Bob's fees in advance, which allows Bob to scam Alice of fees, making it an unsolved problem.

Your solution is absolutely not atomic and suffers from a nothing at stake problem.
